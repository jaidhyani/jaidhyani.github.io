Doc gaps with no PR in flight

  • Horizontal scaling — uvicorn workers, Redis-ize in-process singletons, load test. Source: [Doc T2] "Small-Medium Gaps" row — "30 users, 100+ concurrent, 1-2 weeks. Add uvicorn workers, fix in-process singletons to use Redis. Load test."
  • Multi-provider (OpenAI, Gemini) — flagged as the strategic blocker, not a time-boxed one. Luthien v3.0 deliberately dropped non-Anthropic providers; PR #547 is actively removing remaining OpenAI references. Source: [Doc T2] "Multi-Provider: The Strategic Question" — lists 4 options (convince maintainers, fork, split proxies, accept Anthropic-only).
  • Session resume / replay — Claude Code session format is undocumented/internal. Source: [Doc T2] Large Gaps — "Claude Code's session format is internal/undocumented. Export exists, resume needs format conversion."
  • Lossless session capture — Stop hook + transcript_path merge with API logs. Requires client-side scaffolding (not just proxy work). Source: [Doc T2] Large Gaps — "API calls don't include Claude Code's internal context (skills, CLAUDE.md, file reads). Need Stop hook to capture transcript_path + merge with Luthien's API logs." Phase 3 bullet: "lossless session capture via Stop hook, session export."
  • Lambda deployment — requires persistent Postgres + Redis; Fargate works today, true serverless needs storage-layer rewrite. Source: [Doc T2] Large Gaps.
  • Vector / semantic search — pgvector or external. Source: [Doc T2] Large Gaps — "Postgres tsvector for full-text, pgvector or external for semantic. Standard but significant."

Bottom line

Phase 2 of the roadmap — identity, search, retention, health, metrics, auth separation — is largely already in the PR queue. The outstanding work is Phase 3–4: scaling, multi-provider, resume, and lossless capture. The multi-provider question is strategic, not implementation-time-bounded, and the doc correctly identifies it as the make-or-break item. Lossless capture requires client-side scaffolding (a Claude Code hook + CLI uploader), not just proxy-side work — neither side has an open PR yet.

Points of interest
Lossless capture needs client-side work — the proxy only sees HTTP; Claude Code's skills, CLAUDE.md, file reads, and local tool execution only exist in-process. A Claude Code Stop hook + transcript uploader is the shape proposed in [Doc T2] — but nothing about the receiving endpoint, correlation key, or UI join is designed yet.
OpenCode is the hidden driver — the partner team's use of OpenAI + Gemini via OpenCode is what makes multi-provider "make-or-break." An Anthropic-only Luthien plus a separate proxy for others is a real option the doc names.
Scale target — 30 users, 100+ concurrent, "dozens of agents." This is the stated stress-test profile; no open PR has load-testing or worker-count work.
Aurora Serverless v2 compatibility claim rests on asyncpg + standard Postgres — verified by the doc, not by any PR.
