visual comparison report — legibility & usability analysis

Executive Summary

Two PRs propose different redesigns of the Luthien Proxy policy configuration UI. PR #372 takes a breadth approach — simplifying the policy list, redesigning the landing page, and updating the nav across all pages to match the PBC marketing site. PR #376 takes a depth approach — introducing a three-column Available→Proposed→Active workflow with chain building, credential management, and inline policy examples.

Neither PR is strictly better. They optimize for different user profiles and could be combined.

2. Policy Configuration Page

The core comparison. Both PRs redesign the policy configuration page but take fundamentally different approaches.

PR #372 — Two-Column: List + Sticky Config Panel
PR #376 — Three-Column: Available | Proposed | Active

3. Layout & Legibility

PR #372 — Two-Column
  • Simple mental model: scroll to browse, click to configure in the sticky right panel
  • Policy grouping: "Simple" (4 curated) vs "Advanced" reduces cognitive load for new users
  • Sticky config panel stays visible while scrolling the list — no context switching
  • Generous card sizing: each policy card shows name + full description + config keys
  • Weakness: test panel at the top consumes ~40% of the initial viewport before the policy list even begins
  • Weakness: no way to compare proposed config against active config side-by-side
PR #376 — Three-Column
  • Workflow-driven layout: left-to-right Available→Proposed→Active makes the lifecycle visible at a glance
  • Inline examples: each policy card shows concrete In/Out examples (e.g. "quick→fast", "rm -rf→blocked")
  • Side-by-side comparison: proposed and active configs visible simultaneously
  • Information dense: all three states visible without scrolling on wide screens
  • Weakness: three columns compress the Available column to ~25% width — cards feel cramped
  • Weakness: flat alphabetical list (no Simple/Advanced grouping) — 18 policies to scan
  • Weakness: on narrow screens (<1000px), columns stack vertically and the spatial metaphor breaks down

5. Information Architecture

PR #372 — Progressive Disclosure

Curates the experience by splitting policies into Simple (4 vetted policies) and Advanced (everything else). New users see the most useful policies first. Landing page hides developer reference behind a collapsible section.

Breadth — touches 12 files across nav, landing page, and policy config. Makes the whole gateway UI feel cohesive.

PR #376 — Everything Visible

Shows all policies alphabetically with inline examples showing concrete input→output transformations. Users can immediately see what each policy does without clicking. The three-column layout makes the policy lifecycle spatially explicit.

Depth — touches only 4 files but introduces chain building, credential management, and a fundamentally new workflow model.

6. Design Consistency

  • PR #372 aligns with PBC marketing site: #09090b background, #6b9fff blue accent, Inter + JetBrains Mono fonts, frosted glass nav. Updates nav across all pages for consistency.
  • PR #376 keeps the old nav and design tokens. The three-column layout is well-crafted internally but doesn't match the direction the product design is heading.
  • PR #372's nav.css changes affect every page — merging #376 after #372 would require adapting #376's settings popover to work with the new nav structure.
  • PR #376 places settings in the nav bar (a good UX pattern) while #372 puts it in the page header. The nav approach is arguably better since settings affect the gateway globally, not just this page.

Verdict & Recommendation

Neither PR supersedes the other. They excel in complementary areas:

Merge from #372
  • PBC-aligned nav across all pages
  • Redesigned landing page with progressive disclosure
  • Simple/Advanced policy grouping
  • Accessibility improvements
  • Pydantic schema-driven config forms
Merge from #376
  • Three-column Available→Proposed→Active layout
  • Policy chain building (Single/Chain toggle)
  • Credential source dropdown
  • Inline policy examples (In/Out)
  • Test panels in both Proposed and Active columns
  • Settings in nav (global concern, not page-local)

Ideal approach: Merge PR #372 first for the landing page, nav, and accessibility foundations. Then adapt PR #376's three-column policy workflow + chain mode + credential management to work within #372's design system. The combined result would be a polished, powerful policy management experience that serves both new users (progressive disclosure, curated grouping) and power users (chain building, side-by-side comparison, credential management).
